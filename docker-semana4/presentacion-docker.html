Motivación

¿Qué problema resuelve Docker?

  • “En mi máquina sí funciona” → diferencias de OS, versiones y dependencias
  • Tiempo perdido configurando entornos
  • Replicar el mismo ambiente en equipo / servidor
  • Aislamiento: correr varios servicios sin conflicto (puertos, librerías)

Ejemplos típicos

Node 18 vs 20 · PostgreSQL 15 vs 16 · librerías nativas · herramientas distintas por sistema operativo

Concepto

Virtualización vs Contenedores

Máquinas virtuales (VM)

  • Incluyen un sistema operativo completo
  • Más pesadas (RAM/CPU/Disco)
  • Arranque más lento

Contenedores

  • Comparten el kernel del host
  • Más livianos y rápidos
  • Portables: “empaquetan” app + dependencias
  • Aislamiento ≠ seguridad absoluta
Vocabulario

Conceptos clave (1 minuto)

Build & Run

  • Imagen: plantilla inmutable por capas
  • Contenedor: instancia ejecutándose
  • Dockerfile: receta para construir
  • Registry: repositorio de imágenes

Infra

  • Volumen: persistencia fuera del contenedor
  • Red: comunicación entre contenedores (DNS interno)

Meta: entender cómo se conectan servicios tipo api y db.

Cómo funciona

Anatomía mental de Docker

  • Docker Engine: motor que construye y ejecuta
  • Cliente (CLI): docker ...
  • Daemon: gestiona imágenes, contenedores, redes, volúmenes
  • Build context: carpeta enviada al motor durante docker build

Idea clave

Cuando haces docker build, Docker usa los archivos del “contexto” para crear capas reutilizables.

Flujo mínimo

El flujo mínimo en 6 comandos

  • docker pull traer base
  • docker build construir imagen
  • docker run crear + iniciar contenedor
  • docker logs ver salida
  • docker exec entrar al contenedor
  • docker stop / docker rm apagar/limpiar

Regla de oro

Si algo falla: logs primero. Luego revisa puertos, variables y red.

Demo 1

Correr un contenedor listo

Prueba rápida para confirmar que Docker funciona y entender flags.

# Hello World
docker run --rm hello-world

# Linux mínimo (Alpine) en modo interactivo
docker run -it --rm alpine sh

# Ver contenedores
docker ps
docker ps -a

Aprendizajes

  • --rm elimina el contenedor al terminar
  • -it permite interacción (terminal)
  • ps vs ps -a (activos vs todos)
Dockerfile

Primera receta — Node “Hola Mundo”

Objetivo: construir una imagen que exponga un servidor HTTP sencillo.

FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production
COPY . .
EXPOSE 3000
CMD ["node","server.js"]

Clave

El orden importa para el cache: copiar package*.json antes acelera builds.

Performance

Capas y cache (por qué el orden importa)

  • Cada instrucción del Dockerfile crea una capa
  • Si una capa no cambia, Docker la reutiliza (cache)
  • Patrón recomendado: dependencias primero, luego código

Ejemplo

Cambiar un .js no debería obligar a reinstalar dependencias si el Dockerfile está bien ordenado.

Configuración

Variables de entorno

  • -e KEY=value al ejecutar
  • --env-file .env para cargar varias
  • No “hardcodear” secretos en la imagen
  • Config por ambiente: dev vs prod
# Ejemplo
docker run -p 3000:3000 \
  -e PORT=3000 \
  -e DB_HOST=db \
  mi-api:dev
Networking básico

Puertos: host vs contenedor

  • -p 8080:3000 → host:contenedor
  • Varios contenedores pueden exponer el mismo puerto interno
  • Los conflictos ocurren en el puerto del host

Idea clave

El contenedor puede escuchar siempre en 3000; el host decide el puerto real.

Redes privadas

Comunicación entre contenedores

  • Por defecto existe red bridge
  • Mejor práctica: crear una red por proyecto
  • En la misma red: resolución por nombre (DNS interno)
  • Dentro del contenedor, localhost es el contenedor mismo
# Crear red
docker network create micro-net

# Conectar servicios a la red
docker run -d --name db --network micro-net postgres:16
docker run -it --rm --network micro-net alpine sh
Persistencia

Volúmenes: no perder datos

  • Sin volumen: al borrar contenedor, se pierden datos
  • Con volumen: persistencia fuera del ciclo del contenedor
  • Caso típico: PostgreSQL, Mongo, uploads
# Crear y usar volumen
docker volume create pgdata
docker run -d --name db -v pgdata:/var/lib/postgresql/data postgres:16
Demo 2

Postgres con volumen + red

Objetivo: BD persistente y accesible desde otro contenedor (DNS interno).

1) Red
docker network create micro-net

2) Volumen
docker volume create pgdata

3) Levantar Postgres
docker run -d --name db --network micro-net \
  -e POSTGRES_PASSWORD=postgres \
  -v pgdata:/var/lib/postgresql/data \
  -p 5432:5432 postgres:16

4) Probar conexión desde otro contenedor
docker run -it --rm --network micro-net postgres:16 \
  psql -h db -U postgres
Mantenimiento

Limpieza y buenas prácticas

  • Estado: docker ps, docker images
  • Detener/remover: docker stop, docker rm
  • Revisar disco: docker system df
  • Limpiar recursos no usados: docker system prune (cuidado)
  • Nombrar recursos: --name, --network
Troubleshooting

Errores comunes y diagnóstico

  • Port already in use → cambiar el puerto del host: -p 3001:3000
  • Cannot connect to db → revisar red, hostname (db), variables
  • Permission denied con volúmenes → revisar usuario/ruta
  • Build lento → cache + .dockerignore

Comandos útiles

docker logs -f api
docker exec -it api sh
docker inspect api
Puente a Compose

¿Dónde entra Docker Compose?

  • Describe múltiples servicios en un archivo YAML
  • Define redes/volúmenes/variables de forma declarativa
  • Ideal para microservicios locales (db + api + redis…)
  • Hoy: base conceptual; en el taller lo usamos para orquestar
Comandos base
docker compose up --build
docker compose ps
docker compose logs -f
docker compose down
